## 项目评价：Sigma — Web 矢量图形编辑器

**项目定位**: 类 Figma 的在线矢量编辑器，~175 个源文件，单人项目。Canvas 2D 渲染 + React UI + Yjs 协同。

---

### 一、值得肯定的地方

**1. 架构分层清晰**
`editor/`（纯逻辑内核）与 `view/`（React UI）分离得比较干净。编辑器核心不直接依赖 React，渲染管线、场景图、交互状态都在 editor 层独立运作，这是正确的方向。

**2. 自研 React Reconciler（react-reconciler）**
用 react-reconciler 把 React JSX 桥接到 Canvas Widget 层的 Elem 树上（reconciler.ts），让 UI 覆盖层（选框、变换手柄、吸附线等）可以用声明式写法。这说明对 React 内部原理有深入理解，不是普通前端开发者会碰的东西。

**3. 脏矩形渲染**
surface.ts 里实现了 dirty rect 局部重绘、分帧渲染（TinyQueue 按 Manhattan 距离排优先级）、平移时的 `drawImage` 位图搬移优化。这些都是真实的渲染性能优化手段，不是教程级的全量重绘。

**4. Immut + Yjs 绑定**
自研了一套 Immut 状态管理（patch tracking + structural sharing），然后通过 immut-y.ts 双向绑定到 Yjs CRDT，实现协同编辑的数据同步。这个技术选型很有野心，bridging two reactivity systems 是高难度问题。

**5. 文本排版**
自带 grapheme break + Unicode line break 实现（text-break/），不是简单调 `measureText` 就完事。对 CJK、多语言文本断行有实际考量。

**6. Schema 版本迁移**
migration.ts 有版本化的 schema 迁移系统，按版本号顺序执行 transform，这是正确的工程实践。

**7. 几何数学库**
自研 Matrix、MRect、OBB 以及 bezier 曲线相关计算，矩阵链式操作接口设计合理。

---

### 二、架构和技术问题

**1. 全局单例泛滥，依赖关系不透明**

```
// auto-import.ts 里全局注入了十几个单例
YState, YClients, YUndo, Schema, Matrix, MRect, ...
```

几乎所有核心服务都是全局单例 + auto-import 魔法注入。代码里随处直接访问 `YState.find()`、`YClients.client`，完全没有依赖注入。后果：

- 无法写单元测试（无法 mock 依赖）
- 无法创建多个编辑器实例
- 循环依赖风险高（且难以发现）
- 新开发者打开任意文件不知道符号从哪来

**2. 三套响应式系统并存，心智负担重**

项目同时使用了：

- **MobX**（`@observable`, `@computed`, `autorun`, `reaction`）
- **自研 Signal**（`Signal.create()`, `.hook()`, `.dispatch()`）
- **Immut patch 订阅**（`immut.subscribe()`, `YState.flushPatch$`）

三套系统在同一层代码里交叉使用。比如 `StageViewport` 用 MobX observable，`StageSurface` 用 Signal，`StageScene.hookPatchRender` 用 Immut patch。没有清晰的规则说什么场景用什么。这对维护是个定时炸弹。

**3. Immut 实现名不副实**

叫 "Immut" 但 `set`/`insert`/`delete` 全部是原地修改（mutation）：

```typescript
// immut.ts
set = <T>(keyPath: string, value: T) => {
  const parent = this.get<any>(keyPath, true)
  parent[lastKey] = value // 直接修改原对象
}
```

`next()` 时才做一次浅拷贝。这意味着在 `set` 和 `next` 之间，使用者拿到的引用已经是被修改过的。如果任何地方在此期间读了旧值做比较，会出 bug。这不是 immutable，是 "delayed structural sharing with intermediate mutation"，语义非常脆弱。

**4. Schema 双层架构混乱**

`SchemaService`（schema.ts）和 `YStateService`（y-state.ts）职责高度重叠：

- `Schema.find()` → 底层调 `YState.find()`
- `Schema.schema` 字段几乎没人用（数据在 `YState.immut.state` 里）
- `SchemaService` 里的 `immui` 字段是个空类实例（`new (class {})()`），`addItem`/`removeItem` 等方法调的是这个死代码
- `commitOperation` 直接 `return`，是空实现

这说明 Schema 层经历过一次重构但没清理干净，存在大量死代码和误导性接口。

**5. 历史记录系统半废弃**

```typescript
// history.ts
undo() {
    YUndo.undo()
    return  // ← 后面的逻辑永远不执行
    if (!this.canUndo) return
    ...
}
```

`SchemaHistory` 的 undo/redo 已被 `YUndo` 接管，但旧代码还在。两套 undo 系统并存，`SchemaHistory.commit` 仍在被调用并积累 patches。

**6. 类型安全缺口大**

大量使用 `T<any>()` 强转和 `as any`：

```typescript
const parent = T<S.NodeParent>(forwardRef?.node || getNode(node.parentId))
T<any>(this.currentGeometries)[key] = MULTI_VALUE
```

`T` 只是一个类型转换工具（`<T>(v: any) => v as T`），本质上放弃了类型检查。在一个复杂数据结构的编辑器里，缺乏类型守卫意味着运行时错误只能靠肉眼 debug。

**7. 零测试**

175 个源文件，0 个测试文件。对于涉及矩阵运算、坐标变换、脏矩形计算、schema 迁移这类逻辑密集的代码，没有测试基本等于摸着黑写，重构时毫无安全网。

**8. 服务端是空壳**

```typescript
// index.ts — 完整代码
const server = new Server({ port: 1234, name: 'example-document' })
server.listen()
```

协同编辑的后端只是 hocuspocus 的默认模板，没有鉴权、房间管理、持久化逻辑。

### 五、核心建议

1. **统一响应式方案**: 选 MobX 或 Signal 其中一个，彻底干掉另一个。Immut 的 patch 订阅机制合入选定方案
2. **清理 Schema 双层**: 要么 Schema 服务完全变成 thin wrapper，要么把 YState 降级为纯 IO 层。现在两者互相穿插调用，谁都不完整
3. **加测试，至少覆盖数学层和 Schema 迁移**: Matrix、MRect、OBB、bezier 这些纯函数逻辑测试成本最低但收益最高
4. **收敛全局单例**: 考虑用一个 EditorContext 把所有服务聚合，至少让依赖关系显式化

**总结**: 技术深度对单人项目来说很强——能独立写 react-reconciler bridge、脏矩形渲染、CRDT binding 的人不多。但架构上"先跑通再说"的痕迹太重，多套系统并存、死代码残留、零测试等问题如果不治理，随着复杂度增长会越来越难维护。目前最大的风险不是功能缺失，而是技术债让后续开发速度指数级下降。 协同编辑的后端只是 hocuspocus 的默认模板，没有鉴权、房间管理、持久化逻辑。
