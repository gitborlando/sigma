# Codex 项目 Review

Review 时间：2026-06-22

## 结论

Sigma 现在已经从单一 Web 项目推进成了 pnpm monorepo：`apps/web` 承载编辑器主应用，`packages/*` 拆出了 undo、toolkit、utils、nested-assets 和 API types。整体方向仍然很明确：这是一个以 Canvas 2D 自研渲染、矩阵几何模型、Yjs/Immut 状态桥接为核心的类 Figma 在线矢量编辑器。

我对当前项目的判断是：核心技术路线比普通编辑器 demo 更扎实，monorepo 化也让基础库开始收敛；但 `apps/web` 还没有达到“全仓类型检查可作为质量门禁”的状态，主路径旁边混入了旧 Pixi、mock、实验页面和半成品 UI，导致工程可信度被明显拉低。

如果按潜力和核心实现打分，可以给 7.6/10；如果按当前可维护性、可验证性、生产可用性综合打分，我会给 6.8/10。最值得优先做的是让 `pnpm typecheck` 回绿，然后清理或隔离历史实验入口。

## 主要优点

### 1. 渲染内核方向正确

`StageSurface` 已经有主 canvas / top canvas 分层、dirty rect 收集、局部渲染、平移复用 `drawImage`、按缩放切片渲染等意识。`Elem` 场景树里也统一封装了局部矩阵、全局矩阵、AABB、可见性、事件命中和脏区收集。

这不是简单把 React 状态映射到 SVG/DOM，而是在搭一个真实编辑器的渲染层。

### 2. 几何模型比 x/y/rotation 字段更成熟

节点使用 `Matrix` / `MRect` 作为核心几何表达，`StageViewport` 负责 client/canvas/stage/scene 坐标转换，`StageTransformer` 通过矩阵差量处理移动、缩放、旋转。这个方向适合长期支持 frame 嵌套、局部坐标、旋转缩放和多选变换。

### 3. 状态桥接有工程价值

`YState + Immut + Undo` 是当前最有价值的状态主线：业务侧可以用普通对象路径更新，底层同时保留 Yjs 文档、patch flush 和 undo/redo 能力。`@gitborlando/mobx-undo` 也已经拆成 workspace package，并有最小测试覆盖。

### 4. monorepo 拆包正在起效

`packages/*` 的 `typecheck` 本次全部通过。`nested-assets` 插件、`toolkit` 的拖拽/滚轮/disposer/traverser、`sigma-utils` 的小工具、`mobx-undo` 的状态撤销服务，都比放在 app 内部更容易复用和测试。

### 5. UI 方向符合工具型产品

整体仍是 Header + LeftPanel + Stage + RightPanel 的专业工具布局，Linaria 局部样式、图层面板、属性面板、历史面板、缩放/网格/标尺等控件方向正确，没有变成营销页或普通后台。

## 主要问题

### P0：全仓 typecheck 失败，当前质量门禁不可用

`pnpm typecheck` 在 8.8 秒后失败。packages 都通过，`apps/web` 失败。核心错误包括：

- `apps/web/src/editor/geometry/bezier/bezier.ts:2` 引用不存在的 `../xy`
- `apps/web/src/editor/operate/align.ts:140`、`:144` 访问 `Elem.obb`，但当前 `Elem` 暴露的是 `mrect` / `aabb`
- `apps/web/src/view/editor/stage/point.tsx:2` 引用未安装的 `@pixi/react`
- `apps/web/src/view/editor/stage/point.tsx:4` 引用不存在的 `src/editor/stage/pixi`
- `apps/web/src/editor/utils/mock/polyline.ts:4` 使用不存在的 `ISchema`
- `apps/web/src/editor/utils/mock/polyline.ts:7` 调用已不存在的 `SchemaCreator.client()`
- `apps/web/src/view/pages/test.tsx` 里的实验对象被推断为 `{}`，直接写 `x/y/vx/vy` 触发类型错误

这些错误说明 `apps/web/tsconfig.json` 把实验代码、旧 Pixi 代码和 mock 全部纳入了主类型检查。短期建议：

1. 先让主应用类型检查回绿。
2. 对仍有价值的实验代码迁入 `ai/tasks`、`dev-only` 或单独 tsconfig。
3. 删除或修复旧 Pixi 点编辑入口。
4. 将 `OperateAlign` 从 `elem.obb` 改到当前 `elem.aabb` / `elem.mrect.aabb` 模型。

### P1：协作链路仍未默认启用，UI 和底层能力不一致

`apps/web/src/editor/y-state/y-state.ts:152` 中 `YSync.init(fileId, this.doc)` 仍然被注释。`YSync` 内部有 Hocuspocus provider 和 awareness 逻辑，但默认初始化只创建本地 `Y.Doc`。

同时，协作 UI 已经存在：

- `apps/web/src/view/editor/header/cooperate.tsx:23` 把 avatar 写成字符串字面量 `'createUrlFromSvgString(other.userAvatar)'`
- `apps/web/src/view/editor/header/cooperate.tsx:56` 的协作者列表 map 没有 key

这会让“协作”在产品层看起来存在，但实际默认路径没有远程同步。建议尽快二选一：

- 暂时不做协作：隐藏或降级协作 UI，文档明确说明仅本地 Y.Doc。
- 要做协作：恢复 `YSync.init`，补 provider destroy、连接状态、重连、离线提示、awareness 清理和最小双端同步验证。

### P1：多父级选中后 create frame 可能破坏层级关系

`HandleNode.wrapInFrame()` 只取第一个选中节点的父级作为 `oldParent`：

- `apps/web/src/editor/handle/node.ts:148`
- `apps/web/src/editor/handle/node.ts:154`

如果用户跨 frame 或跨 parent 选中多个节点，代码会尝试从同一个 `oldParent` 移除所有节点。非该父级子节点不会真正从原父级 `childIds` 删除，但随后会被插入新 frame，并设置新的 `parentId`。结果可能是：旧父级仍保留 childId，新节点又在 frame 下，schema 出现父子关系不一致。

建议：

- create frame 前限制选中节点必须同父级。
- 或按原 parent 分组移动，并明确新 frame 的插入策略。
- 至少在命令入口禁用混合父级选择。

### P1：`StageScene` 内部订阅没有跟随编辑器 dispose 释放

`StageScene.subscribe()` 只返回了 `setupElems()` 和 `hookRenderNode()`：

- `apps/web/src/editor/render/scene.ts:28`

但 `hookRenderNode()` 内部又把 autorun 和 patch hook 加到了 `this.disposer`：

- `apps/web/src/editor/render/scene.ts:57`

这个内部 disposer 没有在 `subscribe()` 返回值中被 dispose。编辑器反复进入/退出文件时，可能残留 autorun、YState patch hook 或重复渲染链路。

建议把 `this.disposer.dispose` 纳入 `Disposer.combine(...)`，并在重新 init 前确保清空旧订阅。

### P1：数值输入无法提交 0

`InputNum` 的结束逻辑把 falsy 当成无效值：

- `apps/web/src/view/component/input-num.tsx:63`
- `apps/web/src/view/component/input-num.tsx:71`

这会导致用户把 x、y、width、height、radius、rotation 等值改成 `0` 时，`onEnd` 不触发。对设计编辑器来说，归零是高频且合法的操作。

建议判断 `undefined/null/NaN`，不要用 `if (!finalValue)`。

### P1：图片加载失败会让 Suspense 永久挂起

`ImageService.loadImage()` 只处理 `onload`：

- `apps/web/src/global/service/image.ts:31`
- `apps/web/src/global/service/image.ts:34`

如果图片 404、跨域失败或网络失败，Promise 不 resolve 也不 reject。依赖它的图片填充和图片 picker 可能一直停在 Suspense fallback 或等待状态。

建议补 `onerror`、超时、失败缓存策略，并在画布填充层给出可恢复占位。

### P1：外部服务边界需要明确

`apps/web/src/global/sdk/supabase.ts:4` 直接创建 Supabase client 并暴露 anon key。anon key 放前端不一定错误，但必须依赖严格 RLS、最小权限和表策略验证。

`apps/web/src/global/sdk/cos.ts:6` 直接请求公开 SCF URL 获取 COS 临时凭证。这里需要确认服务端有鉴权、限流、来源校验、路径约束和最小权限，否则上传能力可能被外部滥用。

建议在仓库文档中明确：

- Supabase anon key 是 public key，不是 service role key。
- RLS 策略覆盖 `files` 等表。
- COS 临时凭证只能写允许路径，且服务端会校验用户身份。

### P2：生产路径里混有半成品和实验入口

几个例子：

- `apps/web/src/view/editor/right-panel/operate/picker/image.tsx:8` 点击“更换图片”会直接 throw `not implemented yet`
- `apps/web/src/editor/core/command.ts:160`、`:165` 文件删除/导出命令是空 callback
- `apps/web/src/view/router.tsx:20` 暴露 `/test` 路由，且该页面参与 typecheck 并报错
- `apps/web/src/editor/render/react/reconciler.ts:102` 起多个 React host config 方法仍是 `Function not implemented.`

建议把 dev/test 路由和实验组件从正式 router 与主 tsconfig 中移出；半成品 UI 要么隐藏，要么实现到可用闭环。

### P2：项目规范执行有漂移

AGENTS 要求优先 Linaria、不要用 Tailwind、文件/文件夹 kebab-case。当前存在：

- `apps/web/src/view/app.css:3` `@import 'tailwindcss'`
- `apps/web/vite.config.ts:3`、`:24` 启用 Tailwind 插件
- `apps/web/package.json` 仍依赖 `tailwindcss`、`@tailwindcss/vite`、`tailwind-merge`
- `apps/web/src/view/component/shadcn/ui/avatar-circles.tsx` 使用 Tailwind class
- `apps/web/src/view/assets/editor/RP` 使用大写目录
- `GraphemeBreak.ts`、`LineBreaker.ts`、`UnicodeTrie.ts`、`unicodeData.ts`、`immut copy 2.ts` 不符合 kebab-case

如果这些是有意保留，建议在项目规范中注明例外；否则应统一清理，避免后续贡献者无从判断哪套规范有效。

### P2：核心编辑器测试仍不足

目前只发现一个测试文件：

- `packages/mobx-undo/src/index.test.ts`

但最需要测试保护的编辑器核心仍没有覆盖：

- `Matrix` / `MRect`
- `HitTest`
- `Immut` / `immut-y`
- `YState` path 更新和 patch flush
- `Undo` 与 Yjs/MobX 组合撤销
- `HandleNode.copy/paste/wrapInFrame/reHierarchy`
- `StageTransformer` 的单选/多选变换
- 文本断行和 Unicode grapheme/line break

建议先加小而硬的单元测试，不必一开始追求端到端覆盖。

## 建议路线

### 第一阶段：让质量门禁回绿

优先处理 `apps/web` typecheck：

1. 修复 `bezier.ts` 缺失的 `../xy` 引用，或删除未使用的旧 bezier 文件。
2. 修复 `OperateAlign` 的 `elem.obb` 旧 API。
3. 删除/迁移旧 Pixi `PointComp`。
4. 删除/迁移 mock 和 `/test` 实验页面，或用独立 tsconfig 管理。
5. 保持 `pnpm typecheck` 可在 CI 或 pre-merge 中稳定通过。

### 第二阶段：补直接影响编辑体验的运行时问题

优先级建议：

1. `InputNum` 支持提交 `0`。
2. `wrapInFrame` 禁止或正确处理混合父级选择。
3. `StageScene` 释放内部 disposer。
4. 图片加载补 `onerror` 和失败占位。
5. 半成品按钮隐藏或实现闭环。

### 第三阶段：明确协作产品线

如果要把协作作为核心能力：

- 恢复 `YSync.init`
- 给 provider/awareness 补完整生命周期释放
- UI 展示连接状态和协作者状态
- 最少做两个客户端同文档编辑同步验证

如果近期不做：

- 隐藏协作入口
- 文档改成“本地 Yjs 状态模型，远程协作待启用”

### 第四阶段：补核心测试

推荐先建 `apps/web` 的 vitest，不需要先做浏览器 E2E。第一批测试建议覆盖：

1. `Matrix` / `MRect` 的 transform、invert、AABB。
2. `Immut` 的 set/insert/delete/next/getPatches。
3. `YState` 对 Map/Array 路径的本地更新和 Yjs 同步。
4. `HandleNode.wrapInFrame` 的同父级、混合父级边界。
5. `InputNum` 的 `0` 提交行为。

## 最后判断

Sigma 最值得继续投入的是三块：`StageSurface/Elem` 渲染管线、`Matrix/MRect` 几何模型、`YState + Immut + Undo` 状态主线。它们是设计编辑器真正难的部分。

当前最该收敛的是工程边界：让 `apps/web` typecheck 回绿，隔离实验代码，修掉少量会直接影响编辑体验的运行时问题，并明确协作能力到底是“当前功能”还是“后续方向”。这些做完后，项目会从“优秀内核原型”更接近“可持续推进的编辑器产品”。
