# Codex 项目 Review

Review 时间：2026-07-06

## 结论

Sigma 在最近一轮重构后，比 2026-07-02 的状态更健康。最明显的进步是：编辑器服务已经从模块级单例逐步迁到 `Editor + first-di + Service.effect` 的会话模型，`apps/web` 单独 `typecheck` 仍然通过，`setupConsoleMaxError` 已改成 dev 限制，`YSyncService` 也补上了 provider destroy。

但项目还没有到“放心长期快速迭代”的状态。当前主要风险不是编译炸，而是几个真实产品路径的运行时边界：根 `pnpm typecheck` 仍失败，数值输入不能提交 `0`，图片加载失败会永久等待，跨父级 wrap frame 可能破坏 schema，协作 UI 仍像半成品，属性面板和首页还有可见但未闭环的功能。

如果按核心潜力和架构方向，我会给 8/10；如果按当前产品可靠性、可维护性和交付闭环综合看，我会给 7/10。这个分数比上次更稳，但还不是“可以放开手让用户高频使用”的水平。

## 主要进步

### 1. 编辑器 DI 和生命周期模型明显前进

`apps/web/src/editor/index.ts` 现在通过 `Editor` 管理一个 DI container，服务按需 `resolve`，并统一在 editor dispose 时释放继承 `Service` 的 singleton。相比之前散落的模块级单例，这个方向更适合编辑器会话、文件会话和未来多实例隔离。

`Service.effect()` 的统一清理模型也让 `reaction`、事件监听、Yjs observe、provider 连接这些东西有了共同出口。这个抽象是对的，后续应该继续把 session 边界往这里收敛。

### 2. 主应用类型健康度保持住了

`pnpm typecheck:web` 通过，说明 `apps/web` 当前没有明显 TS 编译级问题。对于一个刚经历 DI、渲染、协作、控制器大重构的编辑器项目，这个结果不错。

根 typecheck 仍失败，但失败点不在主应用，而在 `packages/y-plain` 的 playground/test 类型边界。这比“主编辑器到处报错”健康很多。

### 3. 上次 review 的几个 P0/P1 已经收敛

`apps/web/src/index.ts:9` 已改成：

```ts
setupConsoleMaxError(isDEV)
```

生产路径死循环卡死风险已经降下来了。

`apps/web/src/editor/y-adapter/y-sync.ts:30` 现在在 effect cleanup 中调用 `provider.destroy()`，协作 provider 生命周期比之前干净。

`UndoService` 现在持有自己的 `MobxUndoService` 和 `Y.UndoManager`，不再是旧的模块级 `MobxUndo / YUndo` 全局变量，重复打开编辑器时粘住旧会话的风险也下降了。

### 4. 对齐逻辑已经迁到 matrix 路径

`apps/web/src/editor/workbench/design-panel/align.ts` 现在通过 `MRect.of(mrect).shift(shift)` 后写回 `[node.id, 'matrix']`。这比旧版本写 `x/y` 更符合当前渲染和 transformer 的事实源方向。

这说明项目在逐步往 `matrix + width + height` 的几何事实源收敛。

## 主要问题

### P0：根 `pnpm typecheck` 仍然失败

根脚本 `package.json` 里的 `typecheck` 仍然不能作为合并门禁。

当前失败点：

- `packages/y-plain/playground.ts:155`
- `packages/y-plain/src/index.test.ts:295`

本质问题是 `YPlain` 的强类型 path API 和“运行时动态路径 / 非法路径返回 false”的示例混在了一套编译配置里。比如 playground 里故意演示对非数组路径 insert，但 TS overload 会先把它拦住。

建议：

1. playground 不参与 package typecheck，或单独建 `tsconfig.playground.json`。
2. 给动态路径提供显式 API，例如 `insertDynamic` / `setDynamic`。
3. 测试非法路径时明确使用动态 API 或局部 `as any`，不要污染强类型主 API。

### P1：`InputNum` 无法提交 `0`

`apps/web/src/view/component/input-num.tsx:63`：

```ts
if (currentValue.current) {
```

`apps/web/src/view/component/input-num.tsx:71`：

```ts
if (!finalValue) return
```

这会让用户把坐标、旋转、透明度、圆角等输入改成 `0` 时，`onEnd` 不触发。设计工具里归零是高频操作，这个会直接影响编辑体验。

建议判断 `null`、`undefined`、`Number.isNaN`，不要用 falsy 判断数字合法性。

### P1：图片加载失败会永久等待

`apps/web/src/global/service/image.ts:31` 的 `loadImage()` Promise 只处理 `onload`，没有 `onerror`：

```ts
await new Promise<void>((resolve) => {
```

如果图片 404、跨域失败或网络失败，Promise 不 resolve 也不 reject。依赖它的图片填充、图片 picker 和 Suspense fallback 都可能永久挂住。

建议：

1. 补 `htmlImage.onerror` 并 reject。
2. 给图片加载加超时。
3. 失败后缓存失败态或短期避重，避免反复请求。
4. 画布上给图片节点显示错误占位。

### P1：跨父级 `wrapInFrame` 仍可能破坏 schema 层级

`apps/web/src/editor/controller/node.ts:111` 只取第一个选中节点的父级：

```ts
const oldParent = this.yState.find<S.NodeParent>(selected[0].parentId)
```

随后 `apps/web/src/editor/controller/node.ts:117` 从这个 oldParent 移除所有 selected：

```ts
selected.forEach((node) => this.handleNode.removeChild(oldParent, node))
```

如果用户跨 frame / 跨 parent 多选，非该父级节点不会从真实旧父级删除，却会在 `insertChildAt(frameNode, node)` 后改到新 frame，可能造成旧父级还保留 childId、新 frame 又持有同一个节点的 schema 不一致。

建议：

1. 最小修法：wrap 前检查 selected 是否同父级，不同父级直接 return 或禁用菜单。
2. 长期修法：定义清楚跨父级 wrap 的产品行为，并按 parent 分组处理。
3. 补一个小测试覆盖同父级和跨父级两种边界。

### P1：协作 UI 仍未闭环

底层 `YSyncService` 和 `YAwareService` 已经比之前完整，但 UI 还有明显半成品：

- `apps/web/src/view/editor/header/cooperate.tsx:24` 头像是字符串字面量 `'createUrlFromSvgString(other.userAvatar)'`
- `apps/web/src/view/editor/header/cooperate.tsx:58` 协作者列表 map 没有 key
- `apps/web/src/view/editor/stage/stage.tsx:82` 观察边框只设置 border，没有尺寸，也没有视口跟随逻辑
- `apps/web/src/view/editor/stage/cursor.tsx:13` cursor 列表 map 没 key

建议把协作分成两个阶段：

1. 先补基础可信度：头像、key、连接状态、断线状态、cursor 稳定颜色。
2. 再补观察模式：跟随对方 viewport、退出观察、当前观察对象丢失时自动恢复。

### P1：可见功能仍有直接 throw 或空实现

几个用户能直接点到的例子：

- `apps/web/src/view/editor/right-panel/operate/picker/image.tsx:8` 点击“更换图片”直接 `throw new Error('not implemented yet')`
- `apps/web/src/editor/core/command.ts:174` 文件删除 callback 为空
- `apps/web/src/editor/core/command.ts:178` 文件导出 callback 为空
- `apps/web/src/editor/core/command.ts:91` rename callback 仍是注释

建议：未完成的功能先隐藏，或者明确 disabled。可见但不可用的按钮比没有按钮更伤产品信任。

### P2：填充面板还有半成品和调试输出

`apps/web/src/view/editor/right-panel/operate/fill-item.tsx:92`：

```ts
return null
```

这导致 HEX 输入完全不显示。

`apps/web/src/view/editor/right-panel/operate/fill-item.tsx:110`：

```ts
console.log('value: ', value)
```

属性面板是高频路径，建议把这些半成品清掉。HEX 输入要么恢复，要么布局上别保留它的逻辑；alpha 的 console 应删除。

### P2：颜色 picker 混用 HSL 和 HSV

`apps/web/src/view/editor/right-panel/operate/picker/color-picker.tsx:31` 用 `saturationl()`，这是 HSL saturation；`color-picker.tsx:32` 用 `value()`，随后 `color-picker.tsx:39` 用 `Color.hsv(hue, saturation, value)` 输出。

吸管逻辑在 `color-picker.tsx:222` 取 HSL 的 lightness，却在 `color-picker.tsx:226` 写入 HSV value。

这会导致颜色 picker 的状态和最终颜色不稳定，用户用吸管或拖动时可能出现颜色跳变。

建议统一为 HSV 或 HSL。对于这种基础控件，最好补一组颜色转换单测。

### P2：自定义 React reconciler 仍需要加固

`apps/web/src/editor/render/react/reconciler.ts` 的方向很有趣：用 React 管 canvas widget tree。但当前还有几个危险点：

- `prepareUpdate` 里直接 apply props，`commitUpdate` 为空
- `clearContainer` 遍历 `container.children.forEach` 时 child.destroy 会修改 children 数组
- `createTextInstance`、`scheduleTimeout`、`getCurrentEventPriority` 等多个 host config 方法仍直接 throw

现在 widget 数量少，可能没触发。未来 overlay、控制点、协作光标变多后，这块会成为隐性风险。

建议：

1. `clearContainer` 遍历 children 副本。
2. 对 React 18 当前会调用的方法补安全实现或安全 noop。
3. 给 widget mount/update/unmount 做最小测试或独立运行验证。

### P2：首页仍混入开发入口

`apps/web/src/view/pages/home/header.tsx` 暴露了：

- `/test` 测试页入口
- `Mock页`
- 无 handler 的 `new file` 按钮

`apps/web/src/view/router.tsx` 也把 `/test` 作为正式路由注册。作为开发期入口没问题，但正式产品入口应该收敛到 dev-only，或者至少按 `isDEV` 隐藏。

### P2：外部服务边界需要文档化

`apps/web/src/global/sdk/supabase.ts` 直接使用 Supabase anon key。anon key 放前端本身不一定错，但必须依赖严格 RLS。

`apps/web/src/global/sdk/cos.ts` 通过公开 SCF URL 获取 COS 临时凭证。这里需要确认服务端具备：

- 鉴权
- 限流
- 来源校验
- 路径约束
- 最小权限

建议把这些安全边界写入 README 或 `ai/notes`，明确哪些 key 是 public、哪些权限由服务端/RLS 兜住。

### P2：项目规范和实际实现仍有漂移

AGENTS 指南要求不要用 Tailwind，优先 Linaria；当前仍存在：

- `apps/web/src/view/app.css:3` import Tailwind
- `apps/web/vite.config.ts` 启用 `@tailwindcss/vite`
- `apps/web/src/view/component/shadcn/lib/utils.ts` 使用 `tailwind-merge`
- `apps/web/package.json` 仍有 Tailwind 相关依赖

如果 Tailwind 是历史组件兼容层，建议在规范里写清楚例外范围；否则后续新增代码会不知道该遵循哪一套样式体系。

### P2：测试覆盖仍然偏薄

当前真正的单测主要集中在：

- `packages/y-plain/src/index.test.ts`
- `packages/mobx-undo/src/index.test.ts`

最需要保护的编辑器主链路仍缺测试：

- `Matrix` / `MRect`
- `HitTest`
- `StageTransformer`
- `RenderTree` / widget reconciler
- `YStateService`
- `UndoService`
- `NodeController.wrapInFrame`
- `InputNum`
- 图片加载失败

建议先补小而硬的单测，不必一开始追求 E2E。

## 建议路线

### 第一阶段：让质量门禁真正回绿

1. 修 `packages/y-plain` typecheck。
2. 让根 `pnpm typecheck` 能稳定通过。
3. 后续 PR 至少跑根 typecheck 或 CI gate。

### 第二阶段：修真实编辑路径的高频问题

1. `InputNum` 支持提交 `0`。
2. `ImageService.loadImage` 补 `onerror` 和超时。
3. `wrapInFrame` 禁止或正确处理跨父级选择。
4. 图片 picker 未实现前隐藏“更换图片”。
5. 文件删除/导出/rename 未完成前不要作为可用命令出现。

### 第三阶段：把协作从“能连”变成“可信”

1. 修头像、key、cursor 稳定颜色、连接状态。
2. 完成观察模式，或者先隐藏观察入口。
3. 给 provider status 暴露到 UI。
4. 最少做一次双客户端同文档同步手测记录。

### 第四阶段：补核心测试

优先测试建议：

1. `InputNum`：0、空值、NaN、parser。
2. `ImageService`：onload、onerror、timeout。
3. `wrapInFrame`：同父级、跨父级。
4. `YPlain`：动态路径与非法路径。
5. `MRect/Matrix`：shift、rotate、invert、AABB。
6. `UndoService`：state/client/all 三类历史。

## 与 2026-07-02 Review 的差异

已经改善：

- `setupConsoleMaxError` 已受 `isDEV` 控制。
- `YSyncService` 已 destroy provider。
- Undo 不再依赖模块级全局实例。
- 对齐逻辑已经改为写 matrix。
- `apps/web` 单独 typecheck 仍通过。

仍未解决：

- 根 `pnpm typecheck` 不绿。
- `InputNum` 不能提交 0。
- 图片加载失败会永久等待。
- `wrapInFrame` 跨父级边界不安全。
- 协作 UI、填充面板、首页入口仍有半成品。
- 测试覆盖仍偏薄。

## 最后判断

这轮之后，Sigma 的“编辑器内核”更像一个可继续重构的系统了，而不是一堆单例服务互相引用。`first-di + Service.effect` 是正确方向，渲染和几何主线也依然是项目最有价值的部分。

下一步最应该做的不是继续大重构，而是把几个小但硬的质量口补上：根 typecheck、InputNum、图片失败、wrapInFrame、半成品 UI。它们都不华丽，但会显著提高项目的可合并性和长期推进速度。
