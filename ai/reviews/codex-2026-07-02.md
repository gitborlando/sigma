# Codex 项目 Review

Review 时间：2026-07-02

## 结论

Sigma 仍然是一个有真实技术含量的类 Figma 在线矢量编辑器。它不是普通 CRUD 项目套壳，核心方向很明确：Canvas 2D 自研渲染、矩阵几何模型、Yjs 状态桥接、React 工具型 UI。主应用相比 2026-06-22 的 review 有明显进步，`apps/web` 单独 `typecheck` 已经通过。

但当前状态依然更像“强个人原型”而不是“稳定产品”。最拖后腿的不是架构想法，而是工程闭环：生产路径混入开发期工具、部分可见按钮没有真正接线、Undo / 协作生命周期不够干净、`matrix` 与旧 `x/y/rotation` 双事实源仍在打架、根级质量门禁仍未回绿。

如果只看内核潜力，我会给 8/10；如果按当前可交付性、可靠性、可维护性综合判断，我会给 6.5/10。

## 主要优点

### 1. 主应用 typecheck 已经回绿

这是相比 2026-06-22 最实质的进步。之前 `apps/web` 自身 typecheck 失败，说明主产品代码无法作为稳定门禁；现在 `pnpm typecheck:web` 能通过，至少主应用层面已经收敛了一轮。

根 `pnpm typecheck` 仍失败，但失败点从主应用迁移到了 `packages/y-plain` 的 playground/test 类型边界，这比主应用到处炸要健康得多。

### 2. 渲染层仍然是项目最强部分

`RendererService` / `RenderSurfaceService` / `RenderTreeService` / `Elem` 这条链路有真实编辑器意识：

- 主 canvas / top canvas 分层
- dirty rect 收集与局部重绘
- 滚动画布时复用已有画面
- 缩放时分片全量渲染
- Elem 场景树统一管理 AABB、矩阵、可见性、事件命中

这不是简单把 React 状态映射到 SVG/DOM，而是在认真搭一个 Canvas 编辑器内核。

### 3. 几何模型方向正确

`Matrix` / `MRect` 作为几何事实源是正确方向。`StageViewport` 负责 client/canvas/stage/scene 坐标转换，`StageTransformer` 用矩阵差量处理移动、缩放、旋转，这套模型适合长期支持 frame 嵌套、局部坐标、多选变换和复杂画布。

问题不在方向，而在旧 `x/y/rotation` 字段还没有完全退出运行路径。

### 4. `YPlain` 有工程价值

`@gitborlando/y-plain` 做了几件有价值的事：

- 把 Y.Map / Y.Array 投影成普通对象状态
- 提供 path 级 set / insert / replace / delete
- 做了 unsafe key 防护
- 支持 observeDeep 后生成 plain patch
- 支持事务内可读一致性

这个包如果打磨好，可以成为项目状态层的重要资产。现在的问题是类型 API 和动态运行时语义还没有完全和解。

### 5. 项目有深度复盘意识

`ai/notes` 里关于 Suspense 实例错配、editor lifecycle effects、selection replay source、YState mirror sync 的记录都很有价值。说明作者不是没意识到复杂编辑器会被生命周期和状态回放咬住，而是已经踩过并总结过。

锐评一句：思考深度够，工程落地还没完全跟上。`effect/session` 的边界还主要停在 note，没有真正变成代码结构。

## 主要问题

### P0：生产入口存在页面自毁逻辑

`apps/web/src/index.ts:9` 无条件调用：

```ts
setupConsoleMaxError()
```

而 `packages/sigma-utils/src/max-error.ts:11` 在错误次数达到阈值后会：

```ts
debugger
while (true) {}
```

开发期这可能是“强制停下来修”的狠工具；生产期这是页面卡死器。只要用户遇到一定数量错误，页面就会进入死循环。

建议：

1. 只在 `isDEV` 下启用。
2. 或者改成只打印 / 上报错误，不要 `debugger` 和死循环。
3. 如果确实要保留 query 开关，默认也必须关闭。

### P0：根 `pnpm typecheck` 仍然失败

根脚本 `package.json:13`：

```json
"typecheck": "pnpm -r --if-present typecheck"
```

当前失败在 `packages/y-plain`：

- `packages/y-plain/tsconfig.json:7` 把 `playground.ts` 纳入 include
- `packages/y-plain/playground.ts:155` 演示非法写入 `plain.insert(['document', 'title'], 'Nope')`
- `packages/y-plain/src/index.test.ts:295` 在事务内插入后读取/写入 `['nodes', 0, 'flags']`

本质问题不是实现必错，而是 `YPlain` 的 API 同时想支持“强类型路径”和“运行时动态路径返回 false”。现在 overload 太严格，导致“故意演示非法路径返回 false”的代码也无法编译。

建议：

1. playground 不参与包 typecheck，或单独 tsconfig。
2. 为动态路径提供显式 API，例如 `setDynamic(path, value)` / `insertDynamic(path, value)`。
3. 测试里故意验证非法路径时，明确用 `as YPlainPath` 或动态 API，不要污染强类型主 API。

### P1：对齐按钮可见但没有真正工作

UI 点击在 `apps/web/src/view/editor/right-panel/operate/align.tsx:17`：

```ts
onClick={() => currentAlign.dispatch(type)}
```

但执行 hook 在 `apps/web/src/editor/operate/align.ts:37` 被注释：

```ts
// this.currentAlign.hook(this.autoAlign),
```

所以对齐按钮大概率只是 dispatch，没有消费者。

更糟的是，即使接上，`horizontalAlign` / `verticalAlign` 只写旧字段：

- `apps/web/src/editor/operate/align.ts:135` 写 `[node.id, 'x']`
- `apps/web/src/editor/operate/align.ts:141` 写 `[node.id, 'y']`

当前渲染主要看 `matrix`，所以会出现“数据改了但画布不动”或几何状态不一致。

建议：

1. 接回 `currentAlign.hook(this.autoAlign)`。
2. 对齐写回必须走 `MRect` / `matrix`。
3. 给对齐加最小测试：单选 frame 子节点、多选节点、旋转节点三种场景。

### P1：Undo / MobxUndo / YUndo 的会话生命周期不干净

`apps/web/src/editor/core/undo.ts` 里有模块级全局：

- `MobxUndo`
- `YUndo`

`UndoService.initUndo()` 会创建新的 `Y.UndoManager`，`destroyUndo()` 也写了，但没看到接入编辑器 dispose 流程。

同时 `HandleSelectService` 在构造时通过 `MobxUndo.register('select', this, ...)` 注册 slice，`packages/mobx-undo/src/index.ts` 里没有 unregister 机制。编辑器反复打开/关闭文件时，旧 service 引用、slice listener、历史状态有粘住的风险。

建议：

1. `UndoService.dispose()` 中调用 `destroyUndo()`。
2. `MobxUndo` 要么跟随 editor session 创建，而不是模块级单例；要么提供 clear/unregister/rebase session API。
3. `HandleSelectService` 的注册生命周期要可释放。

### P1：协作 provider 没完整销毁

`apps/web/src/editor/y-adapter/y-sync.ts:18` 创建 `HocuspocusProvider`，但 dispose 只做了：

```ts
provider.off('synced', this.synced)
provider.off('status', this.status)
```

没看到 `provider.destroy()` 或等价关闭连接逻辑。实时协作长连接如果不释放，切文件、重进编辑器、Suspense 重试都可能残留连接或 awareness 状态。

建议：

1. dispose 中 destroy provider。
2. awareness local state 清理和 provider destroy 顺序明确化。
3. UI 展示 provider status，而不是只展示协作者壳。

### P1：`wrapInFrame` 可能破坏跨父级多选

`apps/web/src/editor/controller/node.ts:105` 只取第一个选中节点的父级：

```ts
const oldParent = this.yState.find<S.NodeParent>(selected[0].parentId)
```

但 `apps/web/src/editor/controller/node.ts:111` 会从这个 oldParent 移除所有 selected：

```ts
selected.forEach((node) => this.handleNode.removeChild(oldParent, node))
```

如果用户跨 frame / 跨 parent 多选，非该父级节点不会从真实旧父级移除，却会被插入新 frame 并改 parentId，schema 父子关系就可能不一致。

建议：

1. 最小修法：wrap 前检查所有 selected 同父级，不同父级直接 return 或禁用菜单。
2. 长期修法：按 parent 分组，定义清楚跨父级 wrap 的产品行为。
3. 给该边界补测试。

### P1：数值输入无法提交 `0`

`apps/web/src/view/component/input-num.tsx:63`：

```ts
if (currentValue.current) {
```

`apps/web/src/view/component/input-num.tsx:71`：

```ts
if (!finalValue) return
```

这会让用户把坐标、宽高、旋转、圆角、透明度等改成 `0` 时，`onEnd` 不触发。对设计工具来说，归零是非常高频的合法操作。

建议改成显式判断：

- `null`
- `undefined`
- `Number.isNaN`

不要用 falsy 判断数字合法性。

### P1：图片加载失败会永久挂起

`apps/web/src/global/service/image.ts` 的 `loadImage()` 只处理 `onload`，没有 `onerror`。如果图片 404、跨域失败、网络失败，Promise 不 resolve 也不 reject。

依赖它的地方包括：

- 画布图片填充：`apps/web/src/editor/render/drawer.ts:288`
- 图片 picker：`apps/web/src/view/editor/right-panel/operate/picker/image.tsx:26`

建议：

1. 补 `onerror` reject。
2. 加超时。
3. 失败后缓存失败态，避免反复请求。
4. 画布层提供错误占位。

### P1：`matrix` 与旧 `x/y/rotation` 双事实源仍在打架

当前 schema 里 `NodeBase` 同时包含 `MRect` 和 `S1.OBBInfo`，即同时有：

- `matrix`
- `width` / `height`
- `x` / `y` / `rotation`

而运行时代码仍混用：

- `apps/web/src/editor/handle/node.ts:66` 用 `OBB.fromRect(node, node.rotation)`
- `apps/web/src/editor/handle/node.ts:90` 用 datum 的 `rotation`
- `apps/web/src/editor/operate/align.ts:135` / `:141` 写 `x/y`
- `apps/web/src/view/editor/stage/cursor.tsx:26` 协作光标把位置写进 `x/y`

但渲染和 transformer 主要看 `matrix`。这会导致一类很阴的 bug：属性数据看似变了，画布不动；或者画布动了，旧字段没更新。

建议：

1. 明确唯一事实源：建议 `matrix + width + height`。
2. `x/y/rotation` 只作为派生显示值，不作为写入目标。
3. 逐步删除或迁移 `OBB.fromRect(node, node.rotation)` 这类旧路径。

### P1：协作 UI 仍像半连接状态

协作底层现在已经会在 `SchemaController` 里调用 `YSync.init()`，比旧 review 有进步。但 UI 仍有明显半成品：

- `apps/web/src/view/editor/header/cooperate.tsx:24` 头像是字符串字面量 `'createUrlFromSvgString(other.userAvatar)'`
- `apps/web/src/view/editor/header/cooperate.tsx:58` 协作者列表 map 没 key
- `observingClientId` 设置后只看到边框组件，没有看到视口跟随逻辑
- `apps/web/src/view/editor/stage/cursor.tsx:15` cursor 列表渲染没 key
- cursor 节点写 `x/y`，而 widget 渲染实际看 `matrix`

建议：协作要么隐藏到 dev，要么补完整闭环：头像、光标、选区、连接状态、观察模式、断线状态。

### P2：首页和产品入口仍然 demo 化

首页暴露：

- `测试页`：`apps/web/src/view/pages/home/header.tsx:31`
- `Mock页`：`apps/web/src/view/pages/home/header.tsx:36`
- 无动作的新建文件按钮：`apps/web/src/view/pages/home/header.tsx:38`
- 硬编码封面外链：`apps/web/src/view/pages/home/files.tsx:42`
- 未定义变量 `--bg-content`：`apps/web/src/view/pages/home/files.tsx:67`

这些不一定影响内核，但会严重影响产品可信度。用户一进门看到测试入口和假封面，就会觉得这是 demo。

### P2：填充面板半成品较多

几个例子：

- `apps/web/src/view/editor/right-panel/operate/fill-item.tsx:92` `HexInputComp` 直接 `return null`
- `apps/web/src/view/editor/right-panel/operate/fill-item.tsx:110` 还留着 `console.log('value: ', value)`
- `apps/web/src/view/editor/right-panel/operate/picker/image.tsx:8` 点击“更换图片”直接 throw `not implemented yet`
- `OperateFillService.onAfterSetFills()` 定义了，但很多新增/删除/可见性/透明度操作没有统一接历史记录

建议：可见功能必须闭环；未完成的功能先隐藏，不要让用户点出异常。

### P2：颜色 picker 的颜色模型不严谨

`apps/web/src/view/editor/right-panel/operate/picker/color-picker.tsx` 同时使用：

- `result.saturationl()`
- `result.value()`
- `Color.hsv(hue, saturation, value)`
- 吸管里 `color.hsl().array()` 后把 lightness 塞给 value

HSL 和 HSV 混用会让颜色跳变或不准。设计工具的颜色控件如果不可信，会直接影响用户信任。

建议统一到 HSV 或 HSL，并补一组颜色转换测试。

### P2：自定义 React reconciler 是高级危险品

`apps/web/src/editor/render/react/reconciler.ts` 里有自定义 host config，用 React 渲染 canvas widget elem。方向有趣，但实现还比较薄：

- `prepareUpdate` 里直接 apply props，`commitUpdate` 空实现
- `clearContainer` 遍历 `container.children.forEach` 时 child.destroy 会修改同一个 children 数组
- 多个 host config 方法仍然 `throw new Error('Function not implemented.')`

现在 widget 少，可能没暴露。未来 overlay、光标、选区、控制点变多后，这块很容易残留或顺序错。

建议：

1. `clearContainer` 遍历副本。
2. 未支持的方法要确认 React 18 当前路径不会调用，或实现安全 noop。
3. 给 widget mount/update/unmount 写测试或最小运行验证。

### P2：安全和外部服务边界需要文档化

`apps/web/src/global/sdk/supabase.ts` 直接暴露 Supabase URL 和 anon key。anon key 在前端出现不一定错，但必须依赖严格 RLS。

`apps/web/src/global/sdk/cos.ts` 通过公开函数 URL 获取 COS 临时凭证。这里必须确认服务端有：

- 鉴权
- 限流
- 来源校验
- 路径约束
- 最小权限

否则上传能力可能被滥用。

建议把这些写进项目文档，明确 public key 与服务端权限边界。

### P2：项目规范执行漂移

AGENTS 指南要求不要 Tailwind、文件夹 kebab-case、优先 Linaria。当前仍存在：

- `apps/web/src/view/app.css:3` import Tailwind
- `apps/web/vite.config.ts:24` 启用 Tailwind 插件
- `apps/web/package.json` 里还有 Tailwind 相关依赖
- `apps/web/src/view/pages/home/files.tsx:48` 使用 Tailwind class
- `apps/web/src/view/assets/editor/RP` 大写目录
- `UnicodeTrie.ts`、`LineBreaker.ts`、`GraphemeBreak.ts`、`unicodeData.ts` 不符合 kebab-case

规范不是宗教，但“写一套、做一套”会让后续维护者不知道该信哪套。

### P2：测试覆盖太薄

当前只有两个测试文件：

- `packages/y-plain/src/index.test.ts`
- `packages/mobx-undo/src/index.test.ts`

最需要保护的编辑器核心没有测试：

- `Matrix` / `MRect`
- `HitTest`
- `Renderer` dirty rect
- `StageTransformer`
- `YState`
- `UndoService`
- `wrapInFrame`
- `InputNum`
- 文本断行

自研编辑器内核没有测试，就等于每次重构都在凭手感。

## 建议路线

### 第一阶段：先清 P0

1. `setupConsoleMaxError()` 只允许 dev 启用。
2. 修 `packages/y-plain` typecheck，让根 `pnpm typecheck` 回绿。
3. 给 CI 加根 `pnpm typecheck`，否则门禁只是本地口号。

### 第二阶段：修用户一眼能点到的坏功能

1. 对齐按钮接线，并改成写 `matrix`。
2. `InputNum` 支持提交 0。
3. 图片加载补 `onerror`，图片上传未实现前隐藏按钮。
4. 新建文件按钮要么实现，要么隐藏。
5. 首页测试页 / Mock 页只在 dev 展示。

### 第三阶段：收敛会话生命周期

1. `YSyncService.dispose()` destroy provider。
2. `UndoService.dispose()` destroy `YUndo`。
3. `MobxUndo` 不要作为跨 editor session 的无边界单例。
4. 梳理 `Service` / `effect` / `session` 边界，把 `ai/notes/editor-lifecycle-effects.md` 的思路落到代码。

### 第四阶段：统一几何事实源

1. 明确 `matrix + width + height` 是唯一可写几何事实源。
2. `x/y/rotation` 只作为 UI 派生值。
3. 所有 align、cursor、wrap、datum、old OBB 工具迁到 `MRect` / `Matrix`。

### 第五阶段：补小而硬的测试

先不要追求 E2E 大而全，第一批建议：

1. `InputNum`：0、空值、NaN、parser。
2. `YPlain`：动态路径、非法路径、事务内读写一致性。
3. `wrapInFrame`：同父级、跨父级。
4. `Matrix/MRect`：shift、rotate、transform、aabb。
5. `UndoService`：state/client/all 三类历史，销毁后重建。

## 与 2026-06-22 Review 的差异

- 协作初始化已经在 `SchemaController` 中接上 `YSync.init()`。
- 一些旧 Pixi / mock 类型问题不再是主应用 typecheck 阻塞点。

仍未解决：

- 根 typecheck 仍不绿。
- 半成品 UI 仍多。
- `matrix` 与旧几何字段混用仍在。
- Undo / 协作生命周期仍需要收敛。
- 测试覆盖仍非常薄。

## 最后判断

Sigma 最值得继续投入的是：

1. `RendererService / RenderSurfaceService / Elem` 渲染内核。
2. `Matrix / MRect` 几何模型。
